   and are confirmed application-time features — NO leakage risk.

7. PHASE 3 FEATURE ENGINEERING PLAN — BUREAU TABLE
----------------------------------------------------------------
All features aggregated to SK_ID_CURR grain

A. Coverage & volume features
   FLAG_NO_BUREAU_HISTORY      — 1 if no bureau records (thin-file)
   COUNT_BUREAU_TOTAL          — total number of bureau credits
   COUNT_BUREAU_ACTIVE         — number of active credits
   COUNT_BUREAU_CLOSED         — number of closed credits
   COUNT_BUREAU_BAD_DEBT       — number of bad debt credits
   COUNT_BUREAU_SOLD           — number of sold credits
   RATIO_ACTIVE_TO_TOTAL       — active / total credits

B. Risk flags (mandatory)
   FLAG_HAS_BAD_DEBT_BUREAU    — any bad debt credit ever

7. CONTEXT — WHY VARIANCE MATTERS IN CREDIT RISK
----------------------------------------------------------------

Zero/near-zero variance columns are removed because:

1. No predictive power — cannot distinguish defaulters from
   non-defaulters if everyone has the same value

2. WoE/IV impact — near-constant columns produce near-zero
   IV and screen out in Phase 4 anyway — better to catch
   them explicitly here

3. Model stability — near-constant columns can cause issues
   in logistic regression (separation problems) and inflate
   feature importance in tree models

4. Regulatory audit — a credit risk model should only include
   features with documented predictive justification.
   Near-constant features would raise questions in model
   validation

Key finding: Several FLAG_DOCUMENT_* columns have strong
protective signals despite near-zero variance (e.g.
FLAG_DOCUMENT_13 at -5.32pp). This demonstrates that
near-zero variance does NOT automatically mean no signal —
bad rate analysis is always required before dropping.
